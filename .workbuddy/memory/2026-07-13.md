## 微信小程序 timeout 报错分析

**报错信息：**
```
Error: timeout
    at Function.<anonymous> (WAServiceMainContext.js?t=wechat&v=3.16.1:1)
```

**根因判断：**
- 该错误是裸 `timeout`，堆栈直接指向微信基础库 `WAServiceMainContext.js`（v3.16.1）内部
- 项目的 `cloud.ts` 在 `callFunction` 的 catch 块中会用 `buildCallError()` 包装所有云函数错误，格式为 `[Cloud] callFunction "xxx" failed after Xms (callId): ...`
- 用户看到的错误**没有** callId / 耗时 / 函数名等业务字段，说明它**未经 cloud.ts 包装**
- 结论：错误源不是业务级 `callFunction` 超时（config.timeout 30s/60s），而是微信基础库内部的连接/调度级超时，走了全局 `onError` 通道

**启动时并发的云函数调用（app.tsx useLaunch）：**
1. `CloudService.init()` — 同步初始化
2. `UserService.bootstrap()` — 触发 `checkSession` → 可能 `login` + `userService.getInfo`
3. `AnimationDatasetService.bootstrap()` — 触发 `animationsVersion` → 可能 `listAnimations(snapshot, 60s)`

**排查建议：**
- 看完整日志：裸 timeout 应伴随 `[Cloud] ✖ xxx err Xms` 日志；若有则说明云函数也超时了，若无则确认错误源非 callFunction
- 看触发时机（启动/操作/弱网）
- 看云开发控制台云函数执行日志，确认是否真的执行
- 缓解：错峰启动、关键路径加一次重试、onError 对 timeout 做友好提示
