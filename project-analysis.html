项目概览

这个仓库的真实形态是一个围绕 bvid 统一业务主键、以云函数为唯一数据入口、并由客户端 service 层接管缓存与失效语义的微信小程序系统。它的核心价值不在页面数量，而在把评分、收藏、搜索、投稿、审核和用户态这些业务能力压进一套稳定的数据边界里。

  • 最强证据位于 miniprogram/app.tsx、miniprogram/services/cloud.ts、miniprogram/services/business.ts 与 cloudfunctions/*。
  • 代码和文档都明确要求页面只能调用 service，客户端不允许直接碰数据库或业务云函数细节。
  • 它不是模板式小程序，而是一个带有清晰读写协议和缓存失效策略的 serverless 应用。

架构设计

项目采用“单仓库前端 + 云函数后端”的 serverless 风格模块化单体。真正的边界不是多个服务进程，而是页面必须经过 service 层再进入云函数这一编码纪律。

核心模块

这个仓库的重心在基础设施与业务边界控制模块，而不在页面 JSX 本身。

CloudService miniprogram/services/cloud.ts

统一承接云函数调用、超时、日志、缓存写入、in-flight 复用与 Tag 精准失效，是前后端边界和缓存边界的共同控制点。

Business Services miniprogram/services/business.ts

把动画、评分、收藏、录入、审核这些动作收束成稳定 API，阻止页面依赖云函数 payload 细节。

UserService miniprogram/services/user.ts

负责静默登录、手机号登录、建档兜底、资料更新和统计读取，是用户体验连续性的关键模块。

Request Cache Core miniprogram/utils/requestCacheCore.ts

定义稳定序列化、哈希键、TTL、LRU 和 Tag 失效，是性能与状态一致性的基础设施。

Review Workflow cloudfunctions/animationSubmit/index.js + cloudfunctions/animationReview/index.js

承接投稿、勘误、删除申请和审批落地，同时返回缓存失效所需元信息，是最关键的业务闭环。

独特设计决策

这个项目最有辨识度的地方，是它对小程序最容易失控的边界做了强约束，而不是继续堆页面功能。

01. 统一以 bvid 作为业务主键

这让页面跳转、缓存 Tag 和跨集合关联都共享同一套标识语义，代价是协议迁移必须全链路一致。

02. 页面禁止直接触达数据库或业务云函数

这显著降低了页面复杂度，但把大量正确性责任集中到了 service 层。

03. 缓存被提升为正式协议的一部分

buildCacheTags() 和 buildInvalidationTags() 说明缓存不是附加优化，而是状态一致性设计。

04. 审核返回元信息以驱动精准失效

animationReview 返回 submissionId、targetBvid、submitterOpenid 等字段，直接服务于前端缓存体系。

05. 搜索算法要求前后端双实现同步

这提升了语义一致性，但也增加了每次算法演进的维护成本。

仓库在基础设施设计上已经有明显工程意识，但自动化质量链路和安全封口还没有跟上。

HIGHsetAdmin 接口缺少管理员鉴权

类别: Security

证据: cloudfunctions/userService/index.js 中 setAdmin() 没有校验调用者是否已经是管理员。

影响: 任意已登录用户理论上都可能给自己或他人提权，直接破坏审核和后台管理边界。

MEDIUM搜索候选集被硬编码截断为 200 条

类别: Scalability

证据: cloudfunctions/search/index.js 查询候选集时固定 .limit(200)。

影响: 数据规模上来后，搜索相关性排序只在截断子集上进行，召回完整性会下降。

MEDIUM自动化质量门禁缺失

类别: Operations

证据: 仓库未检测到 GitHub Actions、Jenkinsfile 或其他 CI 配置。

影响: 前后端协议漂移、缓存规则遗漏和云函数漏部署很难在合并前被自动发现。

MEDIUM云环境硬编码在客户端基础设施中

类别: Operations

证据: miniprogram/services/cloud.ts 直接写死 CLOUD_ENV。

影响: 多环境切换和多人协作容易出错，不利于 staging / production 分离。

待验证未知项

最大的未知项不在目录结构，而在微信云开发运行时和真实数据规模下，这些代码假设是否都能成立。

setAdmin 是否被外层权限再次限制

静态代码看不到管理员鉴权，必须通过实际调用链验证，否则应按生产级漏洞处理。

数据库索引是否按文档建议已落地

仓库无法验证云数据库真实索引配置，若缺失索引，搜索、审核和统计性能都会退化。

云函数部署是否总与前端版本同步

文档多次强调手动重部署，说明线上协议错位是现实风险而不是理论风险。

搜索 200 条候选上限在真实数据集中的效果

当前实现可能适合小数据量，但需要用真实样本验证召回质量。

缓存失效规则是否覆盖所有边缘写路径

主路径设计较完整，但组合操作后的状态一致性仍应通过真实回放验证。
