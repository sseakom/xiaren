
# 虾仁宇宙 - 沙雕动画评分 项目分析报告

## 项目概览

这是一个基于 Taro 4 + React 18 + TypeScript 的微信小程序，配套微信云开发（云函数 + 云数据库）。核心体验是为 B 站沙雕/搞笑动画评分、收藏、标记看过，并支持社区录入、勘误、删除申请与管理员审核。

项目的核心约束是**客户端严禁直接操作数据库**，所有业务读写都必须通过云函数调用，并由 `CloudService` 统一封装，实现了精细的请求缓存机制与基于业务 Tag 的缓存失效策略。

## 架构设计

项目采用**前端小程序 + 后端云函数**的 Serverless 架构模式，严格遵循分层设计原则，实现了业务逻辑与数据访问的彻底分离。

### 架构模式

这是一个典型的**分层架构 + 服务化设计**的小程序项目：

1. **表现层**：React 组件 + NutUI，负责 UI 渲染和用户交互
2. **业务服务层**：`services/business.ts` + `services/user.ts`，封装业务逻辑
3. **数据访问层**：`services/cloud.ts`，统一云函数调用、缓存管理、错误处理
4. **基础设施层**：微信云开发（云函数 + 云数据库 + 云存储）

### 执行流程

```
构建:  yarn build:weapp
开发:  yarn dev:weapp
测试:  npx ts-node tests/request-cache.test.ts
部署:  微信开发者工具上传云函数
```

## 核心模块

项目的架构重心在于**统一的云服务封装层**和**精巧的缓存系统**，以及**围绕 bvid 主键的业务模型设计**。

### CloudService - 云函数统一调用层 `miniprogram/services/cloud.ts`

- **职责**：统一封装所有云函数调用，提供请求缓存、超时控制、日志记录、in-flight 请求复用、写后缓存失效
- **关键特性**：
  - 读写分离策略：读请求走缓存，写请求触发失效
  - Tag 机制：每个缓存条目绑定业务标签
  - 自动降级：缓存失败时透明降级为真实请求
- **架构意义**：这是整个前端的核心基础设施，所有云函数调用都必须通过这一层，实现了横切关注点的统一管理

### RequestCache - 本地请求缓存核心 `miniprogram/utils/requestCacheCore.ts`

- **职责**：实现 TTL 过期、LRU 淘汰、Tag 精准失效的缓存引擎
- **关键特性**：
  - 稳定序列化哈希键（用于缓存 key 生成）
  - 支持按 Tag 批量失效
  - 定时清理任务（在 `app.tsx` 中调度）
  - 大小限制 + LRU 淘汰策略
- **架构意义**：这是项目中最具技术深度的模块，大幅降低了云函数调用频率，提升用户体验

### 云函数集 - Serverless 业务逻辑 `cloudfunctions/*`

- **职责**：12 个云函数分别负责不同业务域的逻辑
- **核心云函数**：
  - `listAnimations`：首页列表、排序、筛选
  - `animationsVersion`：动画全量快照版本号
  - `rating`：评分提交与查询
  - `collection`：收藏/看过
  - `animationSubmit`：录入/勘误/删除申请
  - `animationReview`：管理员审核
- **架构意义**：云函数作为后端入口，承担数据读写、业务校验、权限控制的职责

### UserMediaList - 用户列表通用组件 `miniprogram/components/UserMediaList`

- **职责**：统一"我的收藏"与"我的评分"的分页列表壳子
- **架构意义**：体现了组件复用的设计思想，减少重复代码

## 独特设计决策

多个非显而易见的设计决策定义了这个项目的架构风格和开发约束。

### 1. **bvid 作为唯一业务主键**

与传统使用数据库自增 ID 不同，项目选择 B 站视频 ID（bvid）作为全系统的业务主键：
- **收益**：数据天然与 B 站对齐，导入/导出方便，关联关系清晰
- **代价**：需要处理 bvid 唯一性校验，数据库索引需要额外设计
- **证据**：`miniprogram/types/index.ts`、所有云函数都基于 bvid 查询

### 2. **前端绝对禁止直连数据库**

这是项目最严格的约束：
- **收益**：安全性极高，所有数据访问都经过云函数权限校验
- **代价**：开发成本较高，即使简单查询也需要写云函数
- **证据**：`AGENTS.md`、README 中明确禁止，`services/cloud.ts` 是唯一入口

### 3. **基于业务 Tag 的精准缓存失效**

不同于常见的"写后全量清缓存"，项目实现了精细的 Tag 机制：
- **收益**：缓存命中率高，用户体验好
- **代价**：需要仔细设计 Tag 规则和失效策略，容易遗漏
- **证据**：`services/cloud.ts` 中复杂的 `buildInvalidationTags` 和 `buildCacheTags` 函数

### 4. **云函数统一返回格式 + 前端双接口**

- **收益**：`callCloud`（抛出异常）和 `callCloudSafe`（返回 null）适应不同场景
- **证据**：`services/cloud.ts` 中的两个方法

### 5. **搜索完全前置到本地快照**

- **收益**：前端可以直接做离线搜索，避免额外云函数调用
- **代价**：需要同时维护 `fuzzy.ts`、数据集编排和搜索页调用链
- **证据**：`miniprogram/utils/fuzzy.ts`、`miniprogram/services/animationDataset.ts`

## 待验证的未知项

以下是静态分析无法确定、需要实际运行验证的高价值问题。

### 1. **云函数性能与冷启动**

- **问题**：微信云函数的冷启动时间对用户体验影响有多大？
- **为什么重要**：直接影响首屏加载时间和用户留存
- **验证方法**：在微信开发者工具中监控云函数调用耗时，特别是冷启动场景

### 2. **缓存大小在实际使用中的表现**

- **问题**：当前 768KB 的最大缓存限制对真实用户是否足够？LRU 策略是否合理？
- **为什么重要**：如果缓存经常被淘汰，命中率会下降，缓存意义就不大了
- **验证方法**：添加缓存统计埋点，观察真实用户的缓存命中率和淘汰频率

### 3. **搜索算法的相关性质量**

- **问题**：当前的 fuzzy 搜索算法在真实数据上的相关性如何？
- **为什么重要**：搜索是核心功能，质量直接影响用户体验
- **验证方法**：收集真实搜索词和点击数据，评估搜索结果质量

### 4. **审核流程的用户体验**

- **问题**：用户提交审核后，反馈周期多长？驳回理由是否清晰？
- **为什么重要**：影响用户贡献内容的积极性
- **验证方法**：用户调研或可用性测试

这是一个**架构设计非常清晰、代码质量较高**的微信小程序项目。项目的亮点在于：

1. **严格的分层架构**，前端与后端彻底解耦
